# Análise dos workflows 80449231682 e 80449518778

## Workflow 80449231682

A execução aprovou PHP 8.3, PHP 8.4, MySQL 8.4, PostgreSQL 17 e somente contratos estáticos de Docker/Compose.

Ela não executou:

- build e inicialização integral do Playground;
- autenticação RADIUS `Access-Accept`;
- accounting `Accounting-Response`;
- login HTTP no stack Docker;
- instalação nativa equivalente ao CloudPanel.

A ausência ocorreu porque esses jobs haviam sido condicionados ao label `full-validation`, à `main` ou à execução manual. Isso impedia descobrir uma falha de integração antes do merge.

## Workflow 80449518778

Quando a validação completa foi executada, todos os testes PHP e de banco passaram. O CloudPanel nativo também passou. A falha ocorreu exclusivamente no Playground Docker.

O preflight da aplicação confirmou que NAS e credencial existiam. Entretanto, o FreeRADIUS registrou:

```text
server { # from file /etc/freeradius/radiusd.conf
Ignoring "sql"
```

O entrypoint gravava a configuração em:

```text
/etc/freeradius/3.0
```

A imagem oficial `freeradius/freeradius-server:3.2.10` carregava:

```text
/etc/freeradius
```

Consequentemente, o arquivo SQL era renderizado em uma árvore existente, porém inativa. O servidor iniciava com a configuração padrão, não consultava os usuários do RadiusHub e não retornava `Access-Accept`.

## Correções da 1.4.1

- descobre o `radiusd.conf` realmente carregado pelo binário;
- renderiza SQL, queries, clients e virtual server na árvore ativa;
- interrompe o container quando `Ignoring "sql"` aparecer;
- exige evidência do carregamento de `rlm_sql` antes de iniciar;
- aplica a mesma proteção ao instalador nativo;
- executa Docker Playground e CloudPanel em todo Pull Request;
- mantém build de app, web e FreeRADIUS no próprio job integral;
- não gera nem publica artefatos de release durante o CI do PR;
- mantém `push` somente na `main`, evitando a duplicação push + pull_request.
